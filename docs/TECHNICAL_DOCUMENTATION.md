### Why it's structured this way

The folder boundaries map directly to architectural layers. The rule is simple: **dependencies only point inward**. `adapter/` knows about `domain/`; `domain/` knows nothing about `adapter/`. If you can't import something without pointing outward, you've found the right boundary.

- `cmd/api/main.go` is pure wiring — it instantiates all the concrete types and connects them together. No business logic lives here.
- `internal/domain/` has zero imports from outside the standard library (besides `uuid`). It will never need a database driver.
- `internal/adapter/` is where the messy reality lives — SQL queries, Redis commands, HTTP parsing. These are all replaceable without touching the domain.
- `pkg/logger/` is genuinely shared infrastructure with no domain knowledge. If you added a second service, it could use this package.
- `migrations/` are versioned SQL files managed by `golang-migrate`. Running down migrations is a real operation, not an afterthought.

## 2. Hexagonal Architecture Applied

Hexagonal Architecture (also called Ports & Adapters) answers one question: **how do you keep business logic independent of infrastructure?** The answer is interfaces — the domain defines what it needs, and adapters implement those needs.

### The three layers

```
┌─────────────────────────────────────────────────────┐
│                    Adapters (outer)                  │
│  HTTP handlers, Postgres repos, Redis store,        │
│  Event publisher, Outbox worker                     │
│                                                     │
│   ┌─────────────────────────────────────────┐       │
│   │            Use Cases (middle)           │       │
│   │  CreatePayment, GetPayment,             │       │
│   │  ProcessRefund                          │       │
│   │                                         │       │
│   │   ┌─────────────────────────────┐       │       │
│   │   │       Domain (core)         │       │       │
│   │   │  Entities, Value Objects,   │       │       │
│   │   │  Ports (interfaces)         │       │       │
│   │   └─────────────────────────────┘       │       │
│   └─────────────────────────────────────────┘       │
└─────────────────────────────────────────────────────┘
```

### Inbound ports (driving the application)

These are the use case interfaces. HTTP handlers call them. In this project, the use cases are concrete structs, but they're treated as inbound ports — the handlers only know about `CreatePaymentInput` and `CreatePaymentOutput`, never about database internals.

```
HTTP Request → PaymentHandler → CreatePaymentUseCase
```

### Outbound ports (used by the application)

These are the interfaces declared in `internal/domain/port/`. The use cases depend on these — they call methods like `Save`, `Exists`, `Publish` — but they don't know or care what's on the other side.

| Port | Interface | Adapter |
|------|-----------|---------|
| `PaymentRepository` | `port/payment_repository.go` | `adapter/postgres/payment_repository.go` |
| `OutboxRepository` | `port/outbox_repository.go` | `adapter/postgres/outbox_repository.go` |
| `IdempotencyStore` | `port/idempotency_store.go` | `adapter/redis/idempotency_store.go` |
| `EventPublisher` | `port/event_publisher.go` | `adapter/event/publisher.go` |
| `PaymentProvider` | `port/payment_provider.go` | *(not yet implemented — mock only)* |

### How they connect

All the wiring happens in `main.go`. The sequence is:

```go
// Create the concrete adapters
paymentRepo := postgres.NewPaymentRepository(pool)
outboxRepo  := postgres.NewOutboxRepository(pool)
idemStore   := redis.NewIdempotencyStore(client)

// Publisher and Dispatcher share the same buffered channel (size 256).
// The publisher gets the write-end, the dispatcher gets the read-end.
eventCh    := make(chan port.Event, 256)
publisher  := event.NewChannelPublisher(eventCh)
dispatcher := event.NewDispatcher(eventCh)

// Inject into use cases through port interfaces
createUC := usecase.NewCreatePaymentUseCase(
    paymentRepo,  // satisfies port.PaymentRepository
    outboxRepo,   // satisfies port.OutboxRepository
    idemStore,    // satisfies port.IdempotencyStore
    publisher,    // satisfies port.EventPublisher
)

// Inject use cases into HTTP handler
handler := handler.NewPaymentHandler(createUC, getUC, refundUC)
```

If tomorrow you want to swap PostgreSQL for DynamoDB, you write a new `DynamoPaymentRepository` that satisfies `port.PaymentRepository`, change two lines in `main.go`, and nothing else changes. The use cases, domain, and HTTP layer are completely untouched.

## 3. Domain Layer Walkthrough

The domain layer is in `internal/domain/`. Nothing here imports a database driver, HTTP library, or any infrastructure package. This is intentional and enforced.

### Payment entity — the aggregate root

`internal/domain/entity/payment.go` is the central object. All state changes go through it.

Key design decisions:

**All fields are private, accessed via getters.** This isn't Go orthodoxy — it's domain protection. If `status` were public, any adapter could set `payment.Status = "completed"` without going through the state machine. Instead, all state transitions go through `payment.Transition(newStatus)`, which validates the move.

```go
// This is forced — you can't just set status directly
if err := payment.Transition(valueobject.StatusCompleted); err != nil {
    return err // "cannot transition from pending to completed"
}
```

**Two constructors with different contracts:**
- `NewPayment()` — creates a new payment, runs all validations, rejects invalid state
- `ReconstitutPayment()` — rebuilds from database rows, skips validation (the DB is the source of truth)

This distinction matters. If you stored a payment years ago with different validation rules and tried to hydrate it through `NewPayment()`, it might fail. `ReconstitutPayment()` trusts the stored state.

### Transaction entity

`internal/domain/entity/transaction.go` represents a financial operation — a charge, refund, or chargeback. Transactions are immutable once created except for their status. The `Fail()` and `Complete()` methods are the only mutations allowed.

Transactions are owned by a Payment — you get them via `payment.Transactions()`, not directly from the repository.

### Money value object

`internal/domain/valueobject/money.go` encapsulates an `int64` amount in cents plus a `Currency`. Why not just use `int64`?

Because `float64` is dangerous for money. `0.1 + 0.2 != 0.3` in floating point. By storing cents as integers and keeping the currency together with the amount, you avoid two common bugs:

1. Floating point errors corrupting amounts
2. Adding amounts of different currencies (USD to BRL) — `Money.Add()` returns an error if currencies differ

```go
m1, _ := valueobject.NewMoney(9900, "BRL")  // R$ 99.00
m2, _ := valueobject.NewMoney(100, "USD")    // $1.00
result, err := m1.Add(m2) // err: "cannot add BRL and USD"
```

### Payment status — state machine

`internal/domain/valueobject/payment_status.go` implements a state machine through a simple transition table.

```
pending ──────────────┬──────────────▶ failed (terminal)
                      │
                      ▼
                  processing ─────────▶ failed (terminal)
                      │
                      ▼
                  completed ──────────▶ refunded (terminal)
```

The machine is enforced at the domain level — no database trigger, no application-level if-statement scattered across the codebase. One place, one rule:

```go
var validTransitions = map[PaymentStatus][]PaymentStatus{
    StatusPending:    {StatusProcessing, StatusFailed},
    StatusProcessing: {StatusCompleted, StatusFailed},
    StatusCompleted:  {StatusRefunded},
    // failed and refunded are terminal — no outgoing transitions
}
```

If you try to refund a payment that's `failed`, `CanTransitionTo` returns an error immediately, before any database operation happens.

### Supported currencies

`internal/domain/valueobject/currency.go` defines `BRL`, `USD`, and `EUR`. Adding a new currency is one line in that file.

## 4. Use Case Flows

### CreatePayment

**File:** `internal/usecase/create_payment.go`

```
Client → POST /api/v1/payments
          │
          ▼
   PaymentHandler.CreatePayment()
          │  (parse + validate HTTP request)
          ▼
   CreatePaymentUseCase.Execute()
          │
          ├─ 1. Check Redis: Does idempotency key exist?
          │     └─ YES → Fetch payment from DB, return it (no new payment)
          │
          ├─ 2. Create Money value object (validates amount + currency)
          │
          ├─ 3. Create Payment aggregate (NewPayment)
          │     └─ status = pending
          │
          ├─ 4. Save payment to PostgreSQL
          │
          ├─ 5. Save outbox event to PostgreSQL (same DB transaction conceptually)
          │     └─ event_type = "payment.created"
          │
          ├─ 6. Store idempotency key in Redis (TTL: 24h)
          │
          └─ 7. Publish event via channel (best-effort, non-fatal)
                └─ If this fails, outbox worker will retry
```

The outbox save (step 5) and the Redis store (step 6) happen after a successful DB write. If the server crashes between 4 and 5, you'd have a payment without an outbox event — but this window is tiny, and the idempotency key protects against double-charge on retry.

### GetPayment

**File:** `internal/usecase/get_payment.go`

Simple read path. Two methods:
- `ExecuteByID(ctx, uuid)` — used by `GET /api/v1/payments/{id}`
- `ExecuteByExternalID(ctx, externalID)` — available for integrators who want to look up by their own reference

> **Note:** `FindByID` currently returns the payment with a `nil` transactions slice — the postgres adapter doesn't load transactions in the same query, and `TransactionRepository` has no implementation yet. The `transactions` array in the response will always be empty until this is implemented.

### ProcessRefund

**File:** `internal/usecase/process_refund.go`

```
Client → POST /api/v1/payments/{id}/refund
          │
          ▼
   PaymentHandler.ProcessRefund()
          │
          ▼
   ProcessRefundUseCase.Execute()
          │
          ├─ 1. Fetch payment by ID
          │     └─ 404 if not found
          │
          ├─ 2. Validate state transition: completed → refunded
          │     └─ Error if payment is pending, processing, failed, or already refunded
          │
          ├─ 3. Create refund Transaction with reason in metadata
          │
          ├─ 4. Add transaction to payment aggregate
          │
          ├─ 5. Transition payment status to "refunded"
          │
          ├─ 6. Update payment in PostgreSQL
          │
          └─ 7. Save outbox event ("payment.refunded") + publish via channel
```

Note: The refund in this project is a domain operation — it records the intent and transitions state. Actual money movement would happen in a real `PaymentProvider.Refund()` call, which is not yet implemented (the provider port exists, the adapter doesn't).

## 5. The Outbox Pattern

### The problem it solves

Imagine CreatePayment saves the payment to PostgreSQL successfully, then tries to publish an event (say, to notify an order service). The server crashes between those two steps. The payment exists, but the event is lost. The order never gets updated. Money was taken, order never confirmed — that's a serious bug.

The naive fix — "just wrap them in a transaction" — doesn't work because you can't have a distributed transaction that spans PostgreSQL and a message broker (Kafka, RabbitMQ, or even a Go channel).

### How the outbox solves it

```
┌──────────────────────────────────────────────────────┐
│                  PostgreSQL                          │
│                                                      │
│  ┌──────────────┐    ┌──────────────────────────┐    │
│  │   payments   │    │         outbox           │    │
│  │              │    │  id, aggregate_id,       │    │
│  │  (payment    │    │  event_type, payload,    │    │
│  │   record)    │    │  status=pending          │    │
│  └──────────────┘    └────────────┬─────────────┘    │
│         ↑                        │                  │
│  Sequential writes               │                  │
│  (same request, not same txn)    │                  │
└──────────────────────────────────┼──────────────────┘
                                   │
                        ┌──────────▼──────────┐
                        │   OutboxWorker      │
                        │  (polls every 5s)   │
                        └──────────┬──────────┘
                                   │
                          Publish event via channel
                                   │
                        ┌──────────▼──────────┐
                        │   EventDispatcher   │
                        │   (routes to        │
                        │    handlers)        │
                        └─────────────────────┘
```

The outbox row is written right after the payment row, in the same request flow — but as two separate SQL statements, not a single atomic transaction. This means there's a small crash window: if the process dies between `paymentRepo.Save()` and `outboxRepo.Save()`, you'll have a payment without an outbox event. In practice this window is tiny, and the idempotency key prevents a double charge on retry. A future improvement would be to wrap both inserts in a single PostgreSQL transaction to close this gap entirely.

**Outbox status flow:**

```
pending → processed   (happy path)
pending → failed      (after delivery error, increments attempts)
```

**Worker behavior** (`internal/adapter/event/outbox_worker.go`):
- Polls every `OUTBOX_WORKER_INTERVAL_SECONDS` (default: 5s)
- Fetches up to 100 pending events per tick
- Attempts delivery via the channel publisher
- On success: marks `processed`, sets `processed_at`
- On failure: marks `failed`, stores `last_error`, increments `attempts`

### Dual delivery mechanism

There are actually two delivery paths:

1. **Immediate (channel)**: When a use case publishes an event, it goes into a buffered Go channel (size 256). If the dispatcher is keeping up, the event is consumed right away. Fast path, no DB polling involved.

2. **Durable (outbox)**: The same event is also written to the outbox table. The outbox worker guarantees delivery even if the process crashed before the channel was drained, or if the channel was full and the publish was silently dropped.

This means events can be delivered twice — the downstream handlers must be idempotent. That's a known trade-off with at-least-once delivery.

## 6. Idempotency End-to-End

### The problem

A client sends `POST /api/v1/payments`. The network drops after the server processed the request but before the client got the response. The client retries. Without idempotency, the customer gets charged twice.

### The solution

Every payment has an `external_id` — a client-controlled identifier (your order ID, cart ID, whatever). This is your idempotency key.

```
Redis key: payment:create:{external_id}
Value:     {payment_uuid}
TTL:       24 hours (configurable via IDEMPOTENCY_TTL)
```

**On every CreatePayment request:**

```
1. Check Redis for key "payment:create:{external_id}"
   │
   ├─ KEY EXISTS → Fetch payment from PostgreSQL by ID
   │               Return the original payment (no new charge)
   │               HTTP 200 with original response
   │
   └─ KEY MISSING → Process normally
                    After saving to DB: SET key in Redis
                    HTTP 201 with new payment
```

**What this means in practice:**

- Same `external_id`, same amount, same provider: returns the same payment — idempotent
- Same `external_id`, different amount: still returns the original — the first request wins
- Different `external_id`: creates a new payment

The TTL of 24 hours means keys are automatically cleaned up. If a client retries after 25 hours with the same external_id, it would create a new payment. Size your TTL to your retry window.

**Implementation files:**
- Port: `internal/domain/port/idempotency_store.go`
- Adapter: `internal/adapter/redis/idempotency_store.go`
- Usage: `internal/usecase/create_payment.go`

## 7. How to Add a New Payment Provider

The `PaymentProvider` port is already defined at `internal/domain/port/payment_provider.go`. You just need to implement it.

### Step 1: Create the adapter file

```
internal/adapter/stripe/provider.go
```

```go
package stripe

import (
    "context"
    "github.com/ademarthiago/payment-gateway/internal/domain/port"
)

type StripeProvider struct {
    apiKey string
    client *stripe.Client
}

func NewStripeProvider(apiKey string) *StripeProvider {
    return &StripeProvider{
        apiKey: apiKey,
        client: stripe.NewClient(apiKey),
    }
}

// Charge satisfies port.PaymentProvider
func (p *StripeProvider) Charge(ctx context.Context, req port.ProviderRequest) (*port.ProviderResponse, error) {
    params := &stripe.PaymentIntentParams{
        Amount:   stripe.Int64(req.Amount),
        Currency: stripe.String(strings.ToLower(string(req.Currency))),
    }

    pi, err := p.client.PaymentIntents.New(params)
    if err != nil {
        return nil, fmt.Errorf("stripe charge failed: %w", err)
    }

    return &port.ProviderResponse{
        ProviderRef: pi.ID,
        Status:      valueobject.StatusProcessing,
        RawResponse: marshalToJSON(pi),
    }, nil
}

// Refund satisfies port.PaymentProvider
func (p *StripeProvider) Refund(ctx context.Context, tx *entity.Transaction) (*port.ProviderResponse, error) {
    // ... call stripe refunds API
}
```

### Step 2: Update the use case to accept and call the provider

The `PaymentProvider` port is defined but not yet injected into any use case. You'll need to add it:

In `internal/usecase/create_payment.go`:

```go
type CreatePaymentUseCase struct {
    paymentRepo      port.PaymentRepository
    outboxRepo       port.OutboxRepository
    idempotencyStore port.IdempotencyStore
    eventPublisher   port.EventPublisher
    provider         port.PaymentProvider  // ← add this
}

func NewCreatePaymentUseCase(
    paymentRepo port.PaymentRepository,
    outboxRepo port.OutboxRepository,
    idempotencyStore port.IdempotencyStore,
    eventPublisher port.EventPublisher,
    provider port.PaymentProvider,        // ← add this
) *CreatePaymentUseCase { ... }
```

Then call `provider.Charge()` inside `Execute()` after the payment is saved:

```go
resp, err := uc.provider.Charge(ctx, port.ProviderRequest{
    ExternalID:  payment.ExternalID(),
    Amount:      payment.Money().Amount(),
    Currency:    payment.Money().Currency(),
    Description: payment.Description(),
})
if err != nil {
    _ = payment.Transition(valueobject.PaymentStatusFailed)
    _ = uc.paymentRepo.Update(ctx, payment)
    return nil, fmt.Errorf("charge failed: %w", err)
}
_ = payment.Transition(valueobject.PaymentStatusProcessing)
_ = uc.paymentRepo.Update(ctx, payment)
```

### Step 3: Wire it in main.go

```go
var provider port.PaymentProvider

switch os.Getenv("PAYMENT_PROVIDER") {
case "stripe":
    provider = stripe.NewStripeProvider(os.Getenv("STRIPE_API_KEY"))
case "mock":
    provider = mock.NewMockProvider()
default:
    log.Fatal().Msg("unknown PAYMENT_PROVIDER")
}

createUC := usecase.NewCreatePaymentUseCase(
    paymentRepo, outboxRepo, idemStore, publisher, provider,
)
```

### Step 4: Add the env var

In `.env.example`:
```
PAYMENT_PROVIDER=stripe
STRIPE_API_KEY=sk_test_...
```

### Step 5: Add a unit test

Create `internal/adapter/stripe/provider_test.go` with a mock HTTP server or Stripe's test mode. For the use case test, pass a mock that satisfies `port.PaymentProvider` — the rest of the test setup doesn't change.

The HTTP handler and domain layer need no changes.

## 8. How to Add a New Endpoint

Let's say you want to add `GET /api/v1/payments?external_id={id}` — look up by external ID.

### Step 1: Add the use case method (if needed)

If you needed a new use case, create `internal/usecase/your_use_case.go` following the input/output struct pattern.

### Step 2: Add the DTO (if needed)

In `internal/adapter/http/handler/dto.go`:

```go
type GetPaymentByExternalIDRequest struct {
    ExternalID string `json:"external_id"`
}
```

### Step 3: Add the handler method

In `internal/adapter/http/handler/payment.go`:

```go
// @Summary Get payment by external ID
// @Tags payments
// @Produce json
// @Param external_id query string true "External ID"
// @Success 200 {object} GetPaymentResponse
// @Failure 400 {object} ErrorResponse
// @Failure 404 {object} ErrorResponse
// @Router /payments [get]
func (h *PaymentHandler) GetPaymentByExternalID(w http.ResponseWriter, r *http.Request) {
    externalID := r.URL.Query().Get("external_id")
    if externalID == "" {
        writeError(w, http.StatusBadRequest, "external_id is required")
        return
    }

    output, err := h.getPaymentUC.ExecuteByExternalID(r.Context(), externalID)
    if err != nil {
        writeError(w, http.StatusNotFound, "payment not found")
        return
    }

    writeJSON(w, http.StatusOK, toGetPaymentResponse(output))
}
```

### Step 4: Register the route

In `internal/adapter/http/router.go`:

```go
r.Route("/api/v1", func(r chi.Router) {
    r.Post("/payments", paymentHandler.CreatePayment)
    r.Get("/payments", paymentHandler.GetPaymentByExternalID)  // ← add this
    r.Get("/payments/{id}", paymentHandler.GetPayment)
    r.Post("/payments/{id}/refund", paymentHandler.ProcessRefund)
})
```

### Step 5: Regenerate Swagger docs

```bash
make swagger
```

### Step 6: Add an E2E test

Add a test case in `internal/adapter/http/handler/payment_e2e_test.go` following the existing pattern.

## 9. Testing Strategy

### What gets unit tested

**Value objects** (`internal/domain/valueobject/`) — these are pure functions. Test every edge case here. It's cheap and catches subtle state machine bugs.

- `money_test.go`: invalid amounts, currency mismatch on addition
- `payment_status_test.go`: every valid and invalid transition

**Use cases** (`internal/usecase/create_payment_test.go`) — these use mocks from `internal/domain/port/mock/`. The mocks expose `Fn` function pointers, so each test can configure exactly what the "database" and "Redis" return:

```go
func TestCreatePayment_Idempotency(t *testing.T) {
    idemStore := &mock.IdempotencyStoreMock{}
    idemStore.ExistsFn = func(ctx context.Context, key string) (bool, error) {
        return true, nil // simulate: key already exists
    }
    // ... existing payment is returned, no new DB write
}
```

The mocks don't test infrastructure — they test use case *logic*. Does the use case short-circuit on idempotency? Does it return an error on invalid currency? These tests run in milliseconds with no external dependencies.

### What gets E2E tested

**HTTP layer** (`internal/adapter/http/handler/payment_e2e_test.go`) — these tests spin up the real router with `httptest.NewRecorder()` and verify the full HTTP contract: status codes, response body shape, error responses. They still use mocks for the use cases, so they don't hit a real database.

This layer tests: JSON parsing, URL parameter extraction, response serialization, error mapping, and middleware behavior.

### What's not tested (and why)

The PostgreSQL adapters don't have automated tests in the repo — integration tests against a real database are typically run in CI against a test database. The patterns are straightforward enough (pgx queries, no ORM magic) that unit testing them with a mock database driver adds more noise than signal.

## 12. Idempotency Key Usage

The `external_id` field is your idempotency key. Use it to prevent duplicate charges on network retries.

### Rules

- **You choose it.** It should be something meaningful in your system — an order ID, cart ID, or a UUID you generated before the request.
- **It must be unique per payment intent.** If you want to charge for two different orders, use two different `external_id` values.
- **It's valid for 24 hours** (configurable via `IDEMPOTENCY_TTL`). Retries within that window are safe.
- **After 24 hours**, the key expires and a new payment would be created on retry. If that's a concern, use a longer TTL or implement your own check before retrying.

### What happens on retry

```
First request with external_id="order-001":
→ Payment created, id="abc-123", status="pending"
→ Redis key set: payment:create:order-001 = "abc-123"

Second request with external_id="order-001" (same or different body):
→ Redis key found
→ Payment "abc-123" fetched from DB
→ Returned as-is, HTTP 200
→ No new charge, no new DB row
```

### Recommended client implementation

```python
import httpx

def create_payment(order_id, amount_cents, currency):
    payload = {
        "external_id": f"order-{order_id}",
        "amount": amount_cents,
        "currency": currency,
        "provider": "mock",
    }

    for attempt in range(3):
        try:
            resp = httpx.post(
                "http://localhost:8088/api/v1/payments",
                json=payload,
                timeout=10.0
            )
            resp.raise_for_status()
            return resp.json()
        except httpx.TimeoutException:
            if attempt == 2:
                raise
            continue  # retry with same external_id — safe
```

## 14. Payment Lifecycle

### Status definitions

| Status | Meaning |
|--------|---------|
| `pending` | Payment created, awaiting processing |
| `processing` | Being processed by the payment provider |
| `completed` | Successfully charged |
| `failed` | Processing failed (terminal — create a new payment) |
| `refunded` | Fully refunded (terminal) |

### Valid transitions

```
pending ──────────────┬──────────────▶ failed
                      ▼
                  processing ─────────▶ failed
                      ▼
                  completed ──────────▶ refunded
```

### What each transition means for integrators

- **`pending → processing`**: Your payment provider accepted the request and is processing it. Safe to show "processing" in your UI.
- **`processing → completed`**: Money moved. Safe to fulfill the order.
- **`processing → failed`** or **`pending → failed`**: Something went wrong. The error is not retryable with the same payment — create a new one with a new `external_id`.
- **`completed → refunded`**: Refund recorded. The transaction list will show the refund transaction.

**There is no partial refund state** in the current model. A refund transitions the whole payment to `refunded`, regardless of the refund amount.

### A note on production hardening

This project correctly implements the structural patterns (hexagonal arch, outbox, idempotency) but is explicitly a POC. Before production, you'd want to add:

- **Authentication** — the API currently has no auth. Add API key or JWT middleware at the router level.
- **Rate limiting** — chi has middleware for this, or put a gateway in front.
- **Real payment provider** — implement the `PaymentProvider` port for Stripe, PagSeguro, or PIX.
- **Transaction persistence** — `TransactionRepository` is defined as a port but has no postgres implementation yet. `paymentRepo.Update()` currently only persists `status` and `metadata` — the `Transaction` objects added via `AddTransaction()` are **not saved to the DB**. The `transactions` table (migration 000002) and the domain model are ready; the adapter just needs to be written and wired in.
- **Outbox atomicity** — the payment write and the outbox write are two separate SQL statements, not a single transaction. Wrapping them in a `pgx.Tx` would close the crash window between the two.
- **Outbox cleanup** — processed events accumulate forever. Add a job to archive or delete old `processed` rows.
- **Metrics** — Prometheus instrumentation is straightforward to add at the middleware layer.
- **Secrets management** — don't put real credentials in `.env`. Use Vault, AWS Secrets Manager, or equivalent.
- **Partial refunds** — the current model transitions the full payment to `refunded`. If you need partial refunds, the state machine needs a new state and the repository needs to handle it.
